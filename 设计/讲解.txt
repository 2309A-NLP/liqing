在线部分项目讲解提纲
1. config.py —— 配置中心
这一部分讲什么
讲系统里所有外部依赖和默认参数怎么统一管理。

你可以这样说
这个文件负责读取环境变量
负责连接 Redis、Milvus 和大模型服务
负责加载向量模型和重排序模型
所有默认值都集中在这里，避免配置散落到业务代码里
重点说明
为什么要集中管理配置
为什么模型和数据库连接要缓存
为什么用环境变量而不是写死在代码里
2. models.py —— 数据结构定义
这一部分讲什么
讲系统中请求、响应、检索结果、记忆、角色配置这些统一数据结构。

你可以这样说
ChatRequest 定义前端传来的请求格式
ChatResponse 定义后端返回给前端的完整格式
RetrievedDoc 统一描述检索结果
MemoryCandidate 表示一条长期记忆候选项
RoleConfig 定义不同角色的 persona、风格和规则
重点说明
为什么要统一数据结构
为什么响应里不只返回答案，还返回检索和耗时信息
为什么系统支持 assistant、lawyer、doctor 等角色
3. session_memory.py —— 短期记忆
这一部分讲什么
讲如何保存和读取当前会话的最近对话。

你可以这样说
系统把最近几轮对话保存在 Redis 里
每次提问时，都会读取短期历史
然后把这些历史拼成文本，放进 prompt 里
重点说明
短期记忆解决什么问题
为什么它能帮助多轮对话
为什么 Redis 适合做这个事情
4. long_term_memory.py —— 长期记忆
这一部分讲什么
讲系统如何把重要信息沉淀成长期记忆。

你可以这样说
不是所有对话都存长期记忆
系统会先判断内容是否重要
再判断是否重复
最后写入 Redis 和 Milvus
重点说明
长期记忆和短期记忆的区别
什么样的信息值得保存
为什么要做去重
为什么长期记忆既存 Redis 又存 Milvus
5. retrieval.py —— 检索核心
这一部分讲什么
讲系统如何把问题变成检索任务，并找到相关资料。

你可以这样说
这个文件是整个 RAG 的核心之一，主要负责：

问题改写

把用户问题扩展成更适合检索的形式
向量检索

把问题编码成向量
在 Milvus 里找语义相近的内容
BM25 检索

按关键词匹配找相关资料
长期记忆召回

从长期记忆库里找历史重要信息
合并和重排序

把多路结果合并
用 reranker 再排一次
上下文拼接

把检索结果整理成 prompt 可用的上下文
重点说明
为什么要多路检索
向量检索和 BM25 各自擅长什么
rerank 的作用是什么
为什么要控制上下文长度
6. llm.py —— 大模型调用
这一部分讲什么
讲系统怎么调用 DeepSeek 生成答案。

你可以这样说
这个文件封装了大模型调用
支持普通输出和流式输出
普通输出适合一次性返回
流式输出适合前端边生成边显示
重点说明
为什么要把模型调用单独封装
为什么要支持流式返回
为什么 system prompt 要固定
7. api.py —— 在线服务入口
这一部分讲什么
讲整个在线问答流程是怎么串起来的。

你可以这样说
这个文件是系统的总入口，负责：

接收前端请求
读取短期记忆
做问题改写
并行检索知识库、BM25、长期记忆
合并和重排序
拼接 prompt
调用大模型
返回答案
保存短期和长期记忆
路由可以这样讲
/health
健康检查接口
用来确认服务是否正常启动
/chat
非流式问答接口
等模型完整回答后一次性返回
/chat/stream
流式问答接口
适合前端边生成边显示
/session/reset
清空某个会话的短期记忆
/role/presets
返回系统内置角色模板
/memory/list
查看长期记忆记录
/knowledge/random
从知识库里随机抽题，适合演示或测试
重点说明
一次完整问答的主流程
为什么要先检索再生成
为什么要返回检索结果和统计信息
为什么要同时支持流式和非流式
讲解时的总线索
你可以一直围绕这条主线讲：

用户提问 → 读取短期记忆 → 检索知识库和长期记忆 → 拼接 prompt → 调用大模型 → 返回答案 → 保存记忆

这句话基本就是整个项目的核心逻辑。

你可以用的结尾总结
你可以最后这样收尾：

这个项目通过 RAG 把知识检索、短期记忆、长期记忆和大模型生成结合起来，解决了大模型回答不够准确、上下文容易丢失的问题。
同时系统还支持角色化回答和流式输出，比较适合真实业务场景。
