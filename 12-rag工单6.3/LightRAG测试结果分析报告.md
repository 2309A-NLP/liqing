## 二、LightRAG 优势分析

### 1. 综合概括能力强（Q8/Q9/Q10/Q12）
对于需要"理解+归纳"的问题，LightRAG表现优秀：
- Q8（主要业务）：不仅列出业务，还解释了目录销售模式的内涵
- Q10（两家公司区别）：自动组织成5维对比表格，逻辑清晰
- Q12（风险列举）：从招股书多个章节综合提取，分类到6大类20+细项

**原因：** 这类问题需要跨多个chunk综合信息，LightRAG的图谱检索能关联到更多相关实体和关系，比传统RAG的Top-K切片检索覆盖面更广。

### 2. 跨文档检索有效（Q7）
Q7问的是兴图新科的收入，LightRAG从招股说明书1（兴图新科）中准确找到数据，说明图谱入库时正确区分了两份文档的实体。

### 3. 结构化信息提取准确（Q3/Q5）
- Q3（控制关系）：准确提取了人名、持股比例、身份
- Q5（组织结构）：比从原始解析文件中grep的效果更好，说明图谱的实体抽取保留了组织架构信息

## 三、LightRAG 劣势分析

### 1. 精确数值容易出错（Q1/Q2）
**核心问题：** LLM在生成答案时会"编造"或"混淆"数值。

Q1的具体错误：
```
实际数据：发行前5,520万股，拟发≥1,840万股，占发行后≥25%
LightRAG：发行前5,000万股，发1,670万股，占25.04%
```

**原因分析：**
- 图谱中的实体关系可能把不同时间点的股本数据混在一起（历次增资、转增等）
- LLM在生成时选择了"看起来合理"的数值，而不是严格从上下文中提取
- 这是LightRAG的固有缺陷：图谱存储的是LLM抽取的摘要，不是原文，数值精度天然有损

### 2. "反向关系"检索失败（Q4）
Q4问的是"不存在控制关系的关联方"，LightRAG：
- 把5%以上股东（融冰投资等）错误归类为"不存在控制关系的关联方"
- 遗漏了武汉经发（明确的"不存在控制关系的关联方"）

**原因：** 知识图谱存储的是正向关系（"A控制B"、"A持股B"），没有"不存在控制关系"这种反向关系。LightRAG只能基于"关联"关键词检索，无法区分"存在控制关系"和"不存在控制关系"。

### 3. 图表数据提取不完整（Q6）
- 汽车电子增长最快：答对（因为文本中有"仅次于汽车电子"这句话）
- IC卡负增长：没答出（因为这个信息只在图表里，不在文本中）

**原因：** MinerU对图表的OCR提取不完整，图谱构建时图表数据丢失。这是数据源头的问题，不是LightRAG本身的缺陷。

### 4. 数据混淆风险（Q11）
Q11问力源信息的营收，LightRAG返回的数值（7,955/14,884/19,813万元）看起来更像是兴图新科的数据。

**原因：** 两份文档的图谱合并后，实体名称可能有重叠（如"收入"、"万元"等通用实体），导致跨文档检索时混淆。

## 四、优化方案

### 方案1：精确数值问题 → 双引擎互补（优先级：高）

**思路：** 精确数值用传统RAG，综合分析用LightRAG。

```
用户提问 → 判断问题类型
  ├─ 精确数值/表格 → 传统RAG（原文检索，数值准确）
  └─ 综合分析/对比 → LightRAG（图谱检索，覆盖面广）
```

**实施：** 在API层加问题分类器（可以用简单的关键词匹配或LLM分类），自动选择引擎。

### 方案2：实体类型优化（优先级：高）

当前图谱的实体类型可能不够细。优化方向：

```python
# 当前：通用实体类型
entity_types = ["company", "person", "product", "concept", ...]

# 建议：增加金融领域专用类型
entity_types = [
    "company",           # 公司
    "person",            # 人物
    "financial_metric",  # 财务指标（营收、利润、股本...）
    "shareholder",       # 股东
    "related_party",     # 关联方
    "product",           # 产品
    "risk_factor",       # 风险因素
    "project",           # 募投项目
    "department",        # 部门/组织架构
]
```

在 `init_lightrag.py` 的 `addon_params` 中配置更精确的实体类型，让LLM抽取时区分"股本"和"营收"这类容易混淆的数值。

### 方案3：数值精度保障（优先级：高）

在图谱入库时，对数值型实体做特殊处理：

1. **保留原文引用** — 实体描述中保留原始文本片段，不只存摘要
2. **数值校验** — 入库后对关键数值（股本、营收、比例）做交叉验证
3. **时间戳关联** — 数值实体必须关联到具体时间点（2016年/2017年/2018年）

### 方案4：反向关系处理（优先级：中）

对"不存在xxx"这类反向查询，可以：

1. **预处理** — 入库时显式标注"存在控制关系的关联方"和"不存在控制关系的关联方"为两个独立实体
2. **查询改写** — 用户问"不存在控制关系"时，先查"所有关联方"，再排除"存在控制关系的"
3. **传统RAG兜底** — 反向查询自动降级到传统RAG

### 方案5：图表数据补充（优先级：中）

1. MinerU解析时对图表做OCR，生成结构化文本补充到chunks中
2. 或者人工标注图表数据，作为额外的文本块入库
3. 对于Q6这类图表题，可以在chunk中标注"数据来源：图表"

### 方案6：文档隔离检索（优先级：中）

当前两份文档的图谱合并在一起，可能导致跨文档混淆。优化：

1. **在实体上标注来源文档** — 每个实体关联到source_doc字段
2. **查询时按文档过滤** — 如果问题明确提到公司名，只检索该公司的图谱子集
3. **分文档建图谱** — 如果文档差异大，可以建两个独立的LightRAG实例

## 五、总结

### LightRAG 适用场景
| 场景 | 效果 | 原因 |
|------|------|------|
| 综合概括（业务、风险、对比） | ✅ 优秀 | 图谱关联覆盖面广 |
| 跨文档检索 | ✅ 良好 | 实体去重和关联有效 |
| 组织架构/关系查询 | ✅ 良好 | 图谱天然适合关系型信息 |
| 精确数值（股本、营收） | ❌ 差 | LLM抽取丢失精度 |
| 反向关系（"不存在xxx"） | ❌ 差 | 图谱只有正向关系 |
| 图表数据 | ❌ 差 | 依赖OCR质量 |

### 最终建议
**不要用单一引擎，用双引擎互补：**
- 传统RAG：精确数值、原文引用、图表数据
- LightRAG：综合分析、关系推理、跨文档对比

这正是工单要求的"比对RAG、LightRAG对测试问题的检索结果"的目的——找到各自的最优场景，做智能路由。
