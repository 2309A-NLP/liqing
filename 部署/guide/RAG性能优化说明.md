# RAG 性能优化说明

本文档总结了本项目最近几轮对 RAG 检索链路做的性能优化，以及每一项优化的目标和收益。

## 一、优化目标

本次优化主要围绕以下几个方向展开：

1. 降低单次请求的检索耗时
2. 减少不必要的重复计算
3. 缩短送入大模型的上下文长度
4. 增强性能可观测性，便于后续压测和定位瓶颈

---

## 二、优化了什么

### 1. 缩短 embedding 输入长度

原先对查询向量编码时使用较长的输入长度，后来将 `max_length` 从 512 调整为 256。

#### 作用
- 减少 embedding 推理开销
- 对短文本查询更友好
- 通常不会明显损失召回效果

#### 相关位置
- `研发/online/retrieval.py`

---

### 2. 调低 Milvus dense 检索参数

将 dense search 的 `ef` 从 64 调整为 32。

#### 作用
- 降低向量检索开销
- 减少单次搜索耗时
- 对召回结果影响通常有限

#### 相关位置
- `研发/online/retrieval.py`

---

### 3. BM25 索引缓存

原先 BM25 召回每次都要重新构建索引，后来加入了缓存机制。

#### 作用
- 避免每次请求重复构建 BM25 index
- 显著减少文本召回阶段的 CPU 开销
- 数据量越大，收益越明显

#### 相关位置
- `研发/online/retrieval.py`

---

### 4. 检索链路并发化

将以下三个召回步骤并发执行：
- dense 召回
- BM25 召回
- 长期记忆召回

#### 作用
- 缩短总检索时长
- 把串行等待变成并行执行
- 更适合多源检索的 RAG 场景

#### 相关位置
- `研发/online/api.py`

---

### 5. 收缩召回候选数量

对 dense、BM25、memory 的候选数量做了更合理的控制，避免无效扩大候选池。

#### 作用
- 减少重排压力
- 减少后续上下文拼装压力
- 提升整体请求稳定性

#### 相关位置
- `研发/online/api.py`

---

### 6. 上下文拼装做截断控制

`build_context()` 增加了：
- 单文档字符上限
- 总上下文字符上限

#### 作用
- 防止 prompt 过长
- 降低大模型输入 token 数
- 降低 prompt 构造成本
- 有助于减少 LLM 响应延迟

#### 相关位置
- `研发/online/retrieval.py`
- `研发/online/api.py`

---

### 7. Prompt 预览优化

`prompt_preview` 由完整展示改为截断展示。

#### 作用
- 降低无意义的字符串处理开销
- 前端调试更清晰
- 避免把超长 prompt 全量塞进调试字段

#### 相关位置
- `研发/online/api.py`

---

### 8. 增加更细粒度的性能埋点

响应中增加了：
- `timings_ms`
- `retrieval_breakdown`
- `prompt_chars`

#### 作用
- 方便对比优化前后效果
- 便于排查慢点是在 embedding、BM25、dense、rerank 还是 prompt 阶段
- 有利于后续继续做定点优化

#### 相关位置
- `研发/online/api.py`
- `研发/online/models.py`
- `研发/frontend/app.js`

---

### 9. 前端展示性能指标

前端新增了性能统计面板，可以实时看到：
- 总检索耗时
- 向量召回耗时
- BM25 召回耗时
- 记忆召回耗时
- 重排耗时
- Prompt 字符数
- 检索结果数量

#### 作用
- 不用看日志也能直观确认优化效果
- 更方便做人工压测
- 更方便给业务方演示性能变化

#### 相关位置
- `研发/frontend/index.html`
- `研发/frontend/app.js`
- `研发/frontend/styles.css`

---

### 10. 增加基准测试脚本

新增 `优化/benchmark.py`，支持批量请求接口并输出统计信息。

#### 作用
- 快速比较优化前后差异
- 测试平均耗时、P95、最大耗时
- 方便回归验证

#### 相关位置
- `优化/benchmark.py`

---

## 三、整体收益

这几轮优化后，RAG 主链路从“串行 + 重计算 + 长上下文”逐步变成了：

- 查询编码更轻量
- 检索并发执行
- BM25 可缓存
- 上下文可控截断
- 输出可观测

对实际请求的提升通常体现在：
- 检索更快
- prompt 更短
- LLM 输入更少
- 总体延迟更稳定

---

## 四、后续还可以继续优化的方向

如果后面还要继续提速，优先建议：

1. 查询向量缓存
2. 会话历史缓存
3. 长期记忆召回开关
4. 动态 token 截断
5. 更细的 rerank 候选裁剪策略
6. 将性能指标接入正式监控系统

---

如果你需要对外说明，可以直接把本次优化概括为：

> 本项目对 RAG 检索链路进行了并发化、缓存化和上下文瘦身优化，同时增加了性能埋点与前端可视化，整体降低了请求延迟并提升了可观测性。
