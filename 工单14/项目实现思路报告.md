## 一、项目背景与目标

### 1.1 背景
本项目基于 **IMDR / ITMQ 工业多模态数据集**，该数据集具有两个鲜明特征：

1. **不可编辑的图片型 PDF**：1700 个文档以扫描件 / 图片型 PDF 形式呈现，分辨率低、版式不一、内容复杂（专利图纸、技术示意图等），对解析（感知）能力要求极高。
2. **大量交错的图文混合问题**：`questions.jsonl` 含 10,096 个多模态问题，分三类：
   - **纯文本题（Only-Text）**：根据正文文字作答；
   - **纯图像题（Only-Image）**：根据图纸 / 示意图中标号部件的空间位置作答；
   - **图文混合题（Text-Image）**：需同时理解文字描述与技术图纸，进行模块功能或机械原理推理。

### 1.2 核心目标
- **任务一**：成功部署运行 RAGFlow，吃透 `deepdoc` 解析流水线与任务调度机制，产出技术总结文档。
- **任务二**：以 `CN100342976C.pdf`（静电除尘器专利）为标的，构建知识库并**通过调整解析方法、分块策略、向量相似度权重、ReRank 模型等手段（含改代码），使 6 个验收问题的检索精度达到 100%**。
- **核心矛盾**：6 题中 **4 题是第 7 页图纸题**，标准文本 RAG 流水线会丢失图片信息 → 必须打通"图 → 文"的多模态描述链路，这是本项目的胜负手。

## 二、验收问题分析（先看终点，再倒推方案）

| # | 类型 | 问题要点 | 标准答案 | 能力依赖 |
|---|------|---------|---------|---------|
| 1 | 纯文本 | 静电除尘器发明人 | A. P·吉特勒 | OCR 正文 + 文本召回 |
| 2 | 纯文本 | 符合特征的描述 | 管状入口单个圆锥形部分，达外壳直径 80–95%，余采用台阶形式 | OCR 正文 + 文本召回 |
| 3 | 图像 | 第 7 页图中部件 4 相对部件 5 的位置 | 部件 4 位于部件 5 左侧 | **图纸理解（空间关系）** |
| 4 | 图像 | 第 7 页图中尺寸 X1/X2/X3 代表什么 | 配气带孔盘 6,6′,6″ 之间的间隔距离 | **图纸标注理解** |
| 5 | 图像 | 第 7 页图气流方向 (7) 先经过哪个部件 | 先经过 6″，再经过 6′ | **图纸流向推理** |
| 6 | 图像 | 第 7 页图已知外壳直径 D，h1/h2 用来算什么 | 确定配气带孔盘 6,6′,6″ 的位置 | **图纸几何推理** |

**结论**：第 1、2 题靠把扫描件 OCR 准确 + 文本切片/召回调优即可拿下；第 3–6 题必须让**图纸内容被转写成可检索、可推理的文字**（图注 / 多模态描述），否则任何纯文本流水线都不可能命中。整个方案要围绕"**把图片型 PDF 的文字和图纸都不丢地结构化**"展开。

## 三、任务一：RAGFlow 部署 + deepdoc 模块技术分析思路

### 3.1 环境部署方案
1. **拉取源码**：`git clone https://github.com/infiniflow/ragflow.git`，切到稳定 release 分支（避免 nightly 不稳定）。
2. **Docker Compose 一键部署**：进入 `docker/` 目录，使用 `docker compose -f docker-compose.yml up -d` 启动全套依赖：
   - **Elasticsearch / Infinity**（文档与向量存储，二选一，默认 ES）
   - **MySQL**（业务元数据）
   - **Redis**（任务队列 RedisStream + 缓存）
   - **MinIO**（原始文件对象存储）
   - **ragflow-server**（API + 前端）、**task_executor**（解析任务消费者）
3. **镜像选型**：选用**完整版镜像**（含 embedding / deepdoc 模型权重，约 9GB+），而非 slim 版，确保 OCR、版面识别、表格识别模型本地可用。
4. **资源要求**：建议 ≥16GB 内存；若要本地跑 VLM，建议有 GPU 或预留外部多模态 API（见任务二）。
5. **验证**：浏览器访问 `http://localhost:80`，注册登录，确认能新建知识库、上传文件、解析成功。

### 3.2 待回答的三个核心技术问题（分析框架）

> 以下为分析的**目标结论与定位线索**，落地时需在部署好的源码中逐处核对代码行号与最新实现，写入《任务一技术总结文档》。

#### 问题 1：不同 parser_id 的分块策略 + 任务如何触发并入 RedisStream

- **解析模板注册表**：`rag/app/` 目录下每个文件即一种分块模板，对应 `parser_id`：
  - `paper.py` → **Paper**：按论文版式（标题/章节/正文/图表）切，适合结构化长文；
  - `table.py` → **Table**：以表格为中心，结构化抽取行列；
  - `one.py` → **One**：整篇文档作为**单个 chunk**（适合短文档或需要全局上下文的问答）；
  - `knowledge_graph.py` → **Knowledge Graph (KG)**：抽取实体-关系构建知识图谱（GraphRAG）；
  - 其余：`naive(General)`、`manual`、`book`、`laws`、`qa`、`resume`、`presentation`、`picture`、`tag` 等。
- **任务触发链路**（需在代码中核对）：
  1. 前端上传 / 点击解析 → API 层（`api/apps/document_app.py` 的 run/解析接口）；
  2. → `api/db/services/task_service.py` 的 `queue_tasks()`：依据文档页数、`parser_id` **拆分任务**（如大 PDF 按页分段成多个 task），组装任务消息；
  3. → `REDIS_CONN.queue_product(SVR_QUEUE_NAME, message=payload)` 将任务 **写入 RedisStream**（队列名 `rag_flow_svr_queue`）；
  4. 任务执行器以消费组方式从 Stream 拉取，完成"生产者-消费者"解耦。
- **要点**：`parser_id` 决定的是**消费阶段选用哪个 chunker**，而入队逻辑对所有类型基本一致——任务拆分粒度与 `parser_id`、文档页数、是否启用 RAPTOR/GraphRAG 有关。

#### 问题 2：`do_handle_task` 主流程与技术实现

定位：`rag/svr/task_executor.py`。`do_handle_task` 是任务消费的核心，主流程：

1. **取任务与上下文**：解析任务消息，加载 Document / Knowledge Base / 解析配置 / Embedding 模型。
2. **分流处理类型**：
   - `raptor` → 递归聚类摘要（RAPTOR 树）；
   - `graphrag` → 知识图谱构建；
   - 普通解析 → 进入 `build()` 走"分块"。
3. **分块（build）**：`chunker = FACTORY[parser_id]`，调用对应模板的 `chunk()`，内部调用 `deepdoc` 解析器（PDF 走 `RAGFlowPdfParser`）得到带版面坐标的文本块/图块。
4. **向量化（embedding）**：对 chunk 批量调用 embedding 模型生成向量；可选生成关键词、问题（自动 QA）、图注。
5. **写入存储**：`init_kb` 建索引，将 chunk 文本 + 向量 + 位置元数据写入 docStore（ES / Infinity），原图存 MinIO。
6. **进度回写**：通过 `set_progress` 实时更新解析进度与日志，失败重试 / 标记异常。

涉及技术：RedisStream 消费组、批处理并发、向量化、ES/Infinity 混合索引（BM25 + 向量）、断点与进度管理。

#### 问题 3：DeepDoc 深度解析模块（重点：PDF）

- **解析器清单**（`deepdoc/parser/`）：`pdf_parser.py`、`docx_parser.py`、`excel_parser.py`、`ppt_parser.py`、`html_parser.py`、`json_parser.py`、`markdown_parser.py`、`txt_parser.py`、`figure_parser.py`、`docling_parser.py`、`mineru_parser.py`、`paddleocr_parser.py`、`tcadp_parser.py`、`resume/` 等 —— 覆盖 PDF / Word / Excel / PPT / HTML / JSON / Markdown / 纯文本 / 简历等。
- **视觉子模块**（`deepdoc/vision/`）：
  - `ocr.py`：基于 PaddleOCR（转 ONNX）的文字检测 + 识别；
  - `layout_recognizer.py`：版面分析，识别 **标题/正文/图片(figure)/表格/页眉页脚/公式** 等区域；
  - `table_structure_recognizer.py`（TSR）：表格结构（行列单元格）还原；
  - `operators.py` / `postprocess.py`：图像预处理与后处理。
- **PDF 解析技术链（`RAGFlowPdfParser.__call__`）**：
  1. **渲染**：`__images__` 将每页渲染为高分辨率位图（对图片型 PDF 必经此路，因为没有可抽取文本层）；
  2. **OCR**：检测+识别文字框，得到 (文本, 坐标, 置信度)；
  3. **版面识别**：把 OCR 结果归类到版面区域（正文/图/表/标题）；
  4. **表格识别**：对表格区域做 TSR，转成结构化文本/HTML；
  5. **图片区域**：保留 figure 的裁剪图，供后续图注/多模态描述使用（`figure_parser.py` 可调用视觉模型生成描述）；
  6. **文本合并**：按版面与坐标做跨行/跨栏合并、去重，输出带位置的语义块。
- **本项目关键点**：`CN100342976C.pdf` 是**纯图片型扫描专利**，必然走 OCR 全链路；图纸区域被识别为 figure 后，**默认只存图不转文** → 这正是图文信息丢失的根因，任务二须在此处补齐"图→文"。

## 四、任务二：6 题精度 100% 的实现思路

### 4.1 总体策略
分两条战线，先文本后图像，逐题闭环：

- **战线 A（文本题 1、2）**：保证 OCR 质量 + 选对分块 + 召回调优。
- **战线 B（图像题 3–6）**：打通多模态图注链路，让第 7 页图纸内容变成可检索的结构化文字。

### 4.2 战线 A —— 文本题（1、2）实现思路
1. **分块方法选型对比**：对同一文档分别用 **One / Paper / Manual / General** 解析，比较召回效果：
   - 文档较短、答案分散 → **One**（整篇作为上下文，召回最稳，适合保底）；
   - 需要按章节/权利要求结构化 → **Paper / Manual**。
2. **OCR 质量保障**：扫描件分辨率低，必要时提高渲染 DPI、做图像增强（去噪/二值化/放大）预处理，降低 OCR 漏字错字（直接影响"P·吉特勒"等专有名词与"80–95%""台阶形式"等关键短语能否被检索到）。
3. **召回调优**：在「检索测试」中调整
   - **关键词相似度权重 vs 向量相似度权重**（文本专名题适当提高关键词/BM25 权重）；
   - **相似度阈值、Top N**；
   - 选用合适 **Embedding 模型**（中文 + 专业术语，建议 bge-large-zh / bce 等）。
4. ⚠️ **配置同步坑（工单明确提示）**：在"检索测试"页改的权重**不会自动保存**，必须到**聊天助手设置 / 召回算子设置**处同步更新，否则正式问答仍用默认配置。

### 4.3 战线 B —— 图像题（3–6）实现思路（项目核心难点）

第 3–6 题答案全部来自第 7 页图纸的**部件标号空间关系、尺寸标注、气流流向、几何推算**。纯文本/纯向量无法解决，方案如下，按推荐优先级排列：

#### 方案 1（首选）：开启图注 / 多模态描述（VLM Caption）
- **原理**：解析时对 figure 区域调用**多模态视觉模型（VLM / Img2txt）**生成图纸描述，把"部件 4 在部件 5 左侧""6/6′/6″ 为配气带孔盘，X1/X2/X3 为其间隔""气流 7 先经 6″ 再经 6′"等空间信息**写成文字 chunk**，连同原图坐标一起入库。
- **落地**：
  1. 在 RAGFlow 模型设置中配置**多模态模型**（接入 GPT-4o / Qwen-VL / 内置 VLM 等任一可用视觉模型）；
  2. 在 `deepdoc/parser/figure_parser.py` 的图注流程中，使**图片 chunk 走 VLM 描述**（必要时改代码，强制对 figure 启用 caption，而非仅在 picture 模板下启用）；
  3. **Prompt 工程**：给 VLM 的提示词中强约束输出"标号-名称-相对位置-尺寸含义-流向顺序"，并要求**逐对描述标号部件的左右/上下关系**，确保命中第 3–6 题的提问角度。
- **效果预期**：这是让 4 道图像题从 0 分到满分的关键链路。

#### 方案 2（补强）：图文块关联，防止"图文分家"
- 确保图注 chunk 与同页正文 chunk **共享页码/位置元数据**，召回时图、文一起返回，便于图文混合推理与人工核验。
- 在 `pdf_parser` 文本合并阶段，把图纸**邻近的标号说明文字**与图绑定，减少关联错误（工单标题点名的"关联错误"缺陷）。

#### 方案 3（兜底）：第 7 页定向增强
- 若整库自动图注不稳定，可对第 7 页图**单独高清裁剪**，用 VLM 产出**高质量图注文本**，作为补充语料挂载到该文档（保证 6 题可控达标），同时在文档中记录此为定向优化手段。

#### 方案 4：ReRank 精排
- 引入 **Rerank 模型**（如 bge-reranker）对召回结果重排序，把含正确答案的图注/文本块顶到 Top1，配合阈值确保命中。工单"测试设置"亦提示需配置 Rerank 与知识图谱/跨语言搜索选项。

### 4.4 多轮测试与优化闭环（产出物二的核心）
采用"**配置 → 测试 → 归因 → 优化**"循环，每轮记录：

| 轮次 | 解析方法 | 分块策略 | 向量/关键词权重 | Rerank | 是否开图注 | 6 题命中数 | 失败题归因 | 下一步 |
|------|---------|---------|----------------|--------|-----------|-----------|-----------|--------|
| 1 | General | naive | 默认 | 否 | 否 | (基线) | 图像题全错、专名题召回偏 | 换 One + 提关键词权重 |
| 2 | One | one | 关键词↑ | 否 | 否 | … | 文本题改善、图像题仍 0 | 开 VLM 图注 |
| 3 | One/Paper | … | … | 是 | 是 | … | 个别图题描述不全 | 优化 VLM Prompt |
| … | … | … | … | … | … | **6/6** | — | 定稿 |

- 每轮**截图保存检索结果与精度**，归档为多轮测试报告。
- 最终沉淀：调整后的**实现代码 diff**（figure_parser / pdf_parser / 配置）、原因分析、优化方案与效果对照。

## 七、风险与难点

1. **【高】图像题是成败关键**：4/6 题依赖图纸理解，必须打通 VLM 图注，否则上限只有 33%。需提前确认可用的多模态模型（本地 GPU 或外部 API）。
2. **【中】OCR 质量**：低分辨率扫描件易漏字错字，直接影响文本题与专名召回，需图像增强 / 提高 DPI。
3. **【中】配置不同步陷阱**：检索测试页的权重改动不自动保存，必须同步到聊天助手/召回算子设置（工单已明确点名）。
4. **【中】图文关联错误**：图注与正文若元数据不一致会导致召回时图文分家，需保证页码/坐标对齐。
5. **【低】版本差异**：RAGFlow 迭代快，代码行号/接口可能与本报告描述有出入，落地时以部署版本源码为准核对。
