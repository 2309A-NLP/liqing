# 工单15 技术分析文档：优化技术图纸与文本的跨模态检索流程

## 一、RAGFlow 原有检索流程分析

### 1.1 端到端问答流程

RAGFlow 的问答流程经过以下关键阶段：

```
用户提问 → 查询理解 → 混合检索(BM25+向量) → 重排序(Rerank) → 上下文组装 → LLM生成答案
```

**核心代码入口：** `api/db/services/dialog_service.py` 的 `async_chat()` 函数（line 541）。

#### 阶段1：查询理解与重写

- **多轮对话重写**：当开启 `refine_multiturn` 时，`full_question()` 利用LLM将多轮对话合成为独立问题
- **关键词提取**：`keyword_extraction()` 提取关键词并追加到查询
- **跨语言翻译**：`cross_languages()` 将查询翻译为额外语言
- 代码位置：`rag/prompts/generator.py` line 247-305

#### 阶段2：混合检索

由 `rag/nlp/search.py` 中的 `Dealer` 类实现：

1. **BM25全文检索**：`FulltextQueryer.question()` 对查询进行分词、同义词扩展、权重计算，生成 `MatchTextExpr`。检索字段加权：
   - `title_tks^10`、`important_kwd^30`、`question_tks^20`、`content_ltks^2` 等
   - 代码位置：`rag/nlp/query.py` line 27-120

2. **向量相似度检索**：通过嵌入模型将查询编码为向量，使用余弦相似度匹配。代码位置：`search.py` line 180-193

3. **融合策略**：ES/Infinity层面使用 `FusionExpr("weighted_sum", topk, {"weights": "0.05,0.95"})` — 即5% BM25 + 95%向量权重

4. **候选数量**：默认 `topk=1024`，实际窗口约30-64个候选进入重排序

#### 阶段3：重排序（Re-Rank）

三条路径取决于后端引擎：

| 路径 | 条件 | 说明 |
|------|------|------|
| 外部Reranker | 配置了 `rerank_id` | 将 `content_ltks + title_tks + important_kwd` 发送至重排模型（如BGE-Reranker），最终分数 = `tkweight * token_sim + vtweight * reranker_score + rank_feature` |
| ES路径（默认） | 未配reranker + ES后端 | 二次KNN查询获取纯余弦分数，与本地token相似度融合 |
| Infinity路径 | Infinity后端 | 直接使用引擎内置的归一化融合分数 |

代码位置：`search.py` line 513-669

#### 阶段4：上下文组装与提示构建

1. `kb_prompt()`（`rag/prompts/generator.py` line 128）将检索到的chunk格式化为知识块：
   ```
   ID: 0
   ├── Title: 文档名
   └── Content:
   <chunk文本内容>
   ```

2. 系统提示模板（默认）：
   ```
   You are an intelligent assistant. Please summarize the content of the dataset to answer the question.
   Here is the knowledge base:
   {knowledge}
   ```

3. 当 `quote=True` 时，追加引用提示（`citation_prompt.md`），指导LLM插入 `[ID:N]` 标记

4. 代码位置：`dialog_service.py` line 752-769

### 1.2 文档解析流程（PDF多模态处理）

PDF文档的解析入库流程：

```
PDF → OCR(页渲染+PaddleOCR) → 布局分析(ONNX) → 图表提取 → VLM图描述生成 → 分块 → 向量化 → 入库ES
```

#### 1.2.1 布局分析与图表提取

- **布局识别器**：`deepdoc/vision/layout_recognizer.py` — ONNX模型，标签包括：`Text`、`Title`、`Figure`、`Figure caption`、`Table`、`Table caption`、`Header`、`Footer`、`Reference`、`Equation`
- **图表提取**：`deepdoc/parser/pdf_parser.py` 的 `_extract_table_figure()` 方法（line 1206）：
  - 按 `layoutno` 分组收集Figure区域
  - 匹配最近的 Figure caption
  - 裁剪图像区域
  - 输出 `(pil_image, [caption_text])` 元组

#### 1.2.2 VLM图描述生成

- **核心类**：`deepdoc/parser/figure_parser.py` 的 `VisionFigureParser`（line 215）
- **专利文档检测**：`_is_patent_context()` 通过上下文关键词（如"patent number"、"Int. Cl."等）判断是否为专利文档
- **专利图描述提示**（`rag/prompts/vision_llm_patent_figure_describe_prompt_with_context.md`）：
  - 要求输出中文描述
  - 对每个编号部件：描述编号、中文名称、外观、与其他部件的空间位置关系
  - 示例输出：
    ```
    图3表示了一种分散装置的实施例。该图展示了一个落料架(10)的剖面结构。
    在落料架(10)内部，通过紧固机构(11)在一个保护管(12)里设置了一根或多根链条(13)。
    输入管(14)位于落料架(10)的顶部，散料从此进入。
    链条(13)位于保护管(12)之内，保护管(12)通过紧固机构(11)固定在落料架(10)的内侧面上。
    ```
- **并行处理**：`ThreadPoolExecutor(max_workers=10)` 并行调用VLM

#### 1.2.3 分块与入库

- VLM描述文本被 **前置拼接** 到图表的caption文本中（`figure_parser.py` line 306）
- 图像存储至MinIO，`img_id` 字段记录 `{bucket}-{object_key}`
- `doc_type_kwd` 标记为 `"image"` 或 `"table"`
- 文本内容经tokenize后生成 `content_ltks`、`content_sm_ltks`
- 嵌入模型生成向量 `q_{dim}_vec`
- 代码位置：`rag/svr/task_executor.py` line 270-399

### 1.3 关键代码模块定位

| 模块 | 文件 | 核心功能 |
|------|------|---------|
| 查询构建 | `rag/nlp/query.py` | `FulltextQueryer` — 分词、权重、同义词、BM25查询生成 |

| 过滤机制 | `rag/nlp/search.py` line 120 | `get_filters()` — 多后端统一过滤条件构建 |

## 二、多模态检索瓶颈诊断

### 2.1 故障1分析：检索无关

**现象**：用户提问"编号13的部件相对于编号12的部件的位置关系"时，系统返回大量提及"部件"、"编号"但不含图3描述的文本chunk。

**根因分析**：

1. **向量语义匹配的局限**：
   - 查询 "编号13的部件相对于编号12的部件的位置关系" 的向量表示，在语义空间中更接近通用的"部件关系"讨论文本
   - 图3描述chunk的文本（如"链条(13)位于保护管(12)之内"）虽然包含精确答案，但其向量表示可能因描述文本较长、主题更偏向"结构描述"而非"位置关系提问"，导致相似度不够高

2. **BM25关键词匹配的局限**：
   - 关键词 "编号"、"部件"、"位置关系" 在整个专利文档中频繁出现
   - 图3描述chunk中的 "13"、"12" 作为数字关键词权重较低（常见于部件列表）
   - BM25的 `min_match=0.3` 门槛使得大量包含泛关键词的文本chunk通过

3. **融合权重不利**：
   - 融合权重 5% BM25 + 95% 向量 — 向量匹配不精确时，结果主要由向量主导
   - Rerank阶段使用的文本 `content_ltks + title_tks + important_kwd` 也不包含显式的图表类型标记

### 2.2 故障2分析：图文割裂

**现象**：即使检索到了包含图3的页面，系统也无法将查询词与图纸的视觉信息关联，导致LLM缺乏必要的视觉信息来推理答案。

**根因分析**：

1. **提示模板缺乏图感知**：
   - 默认系统提示不包含任何图表处理指引
   - `kb_prompt()` 将所有chunk统一格式化，不区分文本chunk和图表chunk
   - LLM无法识别哪些知识块包含技术图纸描述

2. **图表chunk缺少类型标记**：
   - 知识块格式中只有 `Title` 和 `Content`，没有 `类型` 或 `图表标记`
   - `doc_type_kwd` 字段虽然存在但在上下文组装时被忽略

3. **image_id 未参与答案生成**：
   - `img_id` 在检索结果中作为元数据传递，但 `kb_prompt()` 不包含它
   - 图像不以多模态形式发送给LLM

### 2.3 当前图描述质量评估

基于专利图描述提示（`vision_llm_patent_figure_describe_prompt_with_context.md`），VLM生成的描述质量较高：

- ✅ 包含每个编号部件的编号和中文名称
- ✅ 描述了部件间的空间位置关系（内部/外部/上方/下方/连接）
- ✅ 使用专利文本上下文中的正确中文名称
- ✅ 示例中准确描述了"链条(13)位于保护管(12)之内"

**结论**：图描述本身质量不是瓶颈，问题出在 **检索未能召回** 这些描述chunk，以及 **LLM未被引导** 去参考这些描述。

## 三、优化方案设计

### 3.1 总体架构

在不修改数据库schema、不引入额外模型的前提下，通过三层优化实现跨模态检索增强：

```
用户提问 → [1.查询理解: 图表引用检测] → [2.检索增强: 定向召回+boost] → [3.提示增强: 图表感知指令] → LLM
```

### 3.2 查询理解优化：图表引用检测

**目标**：自动识别问题中的视觉引用（如"图3"、"第11页图示"）。

**实现**：新增 `detect_figure_reference(question)` 函数，使用正则表达式检测：

| 模式 | 示例 | 提取 |
|------|------|------|
| `第(\d+)页.*?图\s*(\d+)` | "第11页图3" | page=11, fig=3 |
| `图\s*(\d+).*?第(\d+)页` | "图3在第11页" | page=11, fig=3 |
| `图\s*(\d+)` | "图3" | fig=3 |
| `第(\d+)页` | "第11页" | page=11 |
| `fig\.?\s*(\d+)` | "fig.3" | fig=3 |
| `figure\s*(\d+)` | "figure 3" | fig=3 |

**返回值**：`{"page_num": 11, "figure_num": 3}` 或 `None`

**位置**：`rag/nlp/query.py` 新增函数

### 3.3 检索增强：定向召回与Re-Rank Boost

**目标**：确保图描述chunk在涉及图表的查询中被优先召回。

**方案A — 定向检索**（主要方案）：

在 `Dealer.retrieval()` 中，当检测到图表引用时：

1. 构造额外的ES/Infinity过滤查询：
   - `doc_id` 匹配当前知识库文档
   - `doc_type_kwd = "image"` — 精确过滤图表类型chunk
   - 页面过滤：在Python层对 `page_num_int` 进行匹配
2. 将定向检索结果合并到候选集，给予高similarity boost

**方案B — Re-Rank Boost**（辅助方案）：

在rerank计算最终分数时：
- 对 `doc_type_kwd == "image"` 且页码匹配的chunk，在最终similarity上加 `figure_boost_weight`（default: 0.3）

**过滤实现**：
- `doc_type_kwd` 过滤通过在 `get_filters()` 中添加该字段支持，利用已有的多后端统一过滤机制
- `page_num_int` 过滤在Python层post-filter实现（因Infinity中该字段存储格式特殊）

**配置项**：
- `figure_boost_enabled` (bool, default: True)
- `figure_boost_weight` (float, default: 0.3)

### 3.4 提示优化：图表感知指令

**目标**：当检索到图描述chunk时，引导LLM优先参考图纸描述回答。

**实现**：

1. **图表标记**：在 `kb_prompt()` 中，当 `doc_type_kwd == "image"` 时，在Content前加上 `[图表描述]` 标记

2. **感知提示注入**：当检索结果中存在图表chunk时，在系统提示末尾追加：
   ```
   注意：以上知识库中包含技术图纸/图表的描述（标记为[图表描述]）。
   图纸描述中包含了部件编号、名称和空间位置关系的详细信息。
   当问题涉及特定部件的位置关系、归属或流程时，请优先参考[图表描述]中的信息。
   部件编号在图纸描述中对应具体的结构组件。
   ```

3. **配置项**：`figure_aware_prompt` (str, default: 内置模板) — 允许用户自定义图表感知提示

### 3.5 兼容性与配置

所有优化通过配置开关控制，确保向后兼容：

| 配置项 | 位置 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| `figure_boost_enabled` | prompt_config | bool | True | 是否启用图表检索增强 |
| `figure_boost_weight` | prompt_config | float | 0.3 | 图表chunk的rerank boost权重 |
| `figure_aware_prompt` | prompt_config | str | 内置模板 | 自定义图表感知提示文本 |

**向后兼容保证**：
- 所有新逻辑仅在检测到图表引用时才注入
- 纯文本查询不受影响
- 配置开关可一键关闭所有优化
