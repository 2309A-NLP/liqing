# 工单15 测试与优化报告

## 一、优化概述

### 1.1 优化目标

针对专利文档 CN100347506C.pdf 的6个测试问题，优化RAGFlow在技术图纸场景下的跨模态检索能力。目标：6个问题准确率 ≥ 80%（验收目标100%）。

### 1.2 优化前问题分析

| 问题 | 类型 | 预期故障 |
|------|------|---------|
| Q1 | 纯文本 | 通常可正确回答 |
| Q2 | 纯文本 | 通常可正确回答 |
| Q3 | 图文关联 | 检索无关：向量/BM25返回泛"部件"文本chunk，遗漏图3描述 |
| Q4 | 图文关联 | 图文割裂：即使检索到图描述，LLM未被引导参考 |
| Q5 | 图文关联 | 检索无关：查询"散料从部件14进入"匹配到工艺描述而非图描述 |
| Q6 | 图文关联 | 图文割裂："调整链条位置"需要结合图3描述推理 |

**根因总结：**
1. 检索阶段：图描述chunk虽然已入库，但纯文本向量/BM25检索无法有效召回（向量语义偏移 + 数字关键词权重低）
2. 提示阶段：系统prompt不区分文本chunk和图表chunk，LLM缺乏图文协同分析引导

### 1.3 优化方案

三层优化：

| 层 | 优化内容 | 影响文件 |
|----|---------|---------|
| 查询理解 | 正则检测"图3"、"第11页"等引用模式 | `rag/nlp/query.py` |
| 检索增强 | 定向召回image类型chunk + rerank boost | `rag/nlp/search.py` |
| 提示增强 | 图表标记[图表描述] + 图表感知prompt | `generator.py`, `dialog_service.py` |

### 2.2 检索增强 — 定向召回 + Boost

**文件：** `rag/nlp/search.py`

#### 变更1：`get_filters()` 增加 `doc_type_kwd` 支持

```python
# 原：只支持 knowledge_graph_kwd, available_int, entity_kwd 等
# 新：增加 doc_type_kwd — 利用已有的多后端统一过滤机制
for key in [..., "doc_type_kwd"]:
```

效果：可在ES/Infinity/OceanBase中统一过滤 `doc_type_kwd = "image"`。

#### 变更2：`Dealer.retrieval()` 增加 `prompt_config` 参数

```python
async def retrieval(self, ..., prompt_config: dict | None = None):
```

#### 变更3：图表定向检索

在主检索之后、rerank之前：

```python
if figure_boost_enabled and question:
    figure_ref = detect_figure_reference(question)
if figure_ref:
    # 构造定向查询：doc_type_kwd = "image"
    figure_req = {"kb_ids": kb_ids, "doc_type_kwd": "image", ...}
    fig_sres = await self.search(figure_req, ...)
    # Python层过滤 page_num_int 匹配
    # 合并到主候选集
```

**关键设计决策：**
- 不修改ES schema，利用已有的 `doc_type_kwd` keyword字段过滤
- `page_num_int` 在Python层post-filter（因Infinity中该字段有hex编码特殊处理）
- 定向检索的chunk合并到主候选集，走正常rerank流程

#### 变更4：Rerank后Boost

```python
if figure_ref and figure_boost_weight > 0:
    for idx in range(len(sres.ids)):
        chunk = sres.field.get(sres.ids[idx], {})
        if chunk.get("doc_type_kwd") != "image":
            continue
        if target_page and target_page not in chunk.get("page_num_int", []):
            continue
        sim_np[idx] += figure_boost_weight  # 默认+0.3
```

**效果：** 图描述chunk的最终相似度分数提高0.3，确保在排序中优先出现。

## 五、已知限制与后续优化

### 5.1 当前限制

1. **依赖VLM图描述质量**：如果VLM未正确识别部件编号或空间关系，检索优化无法弥补
2. **正则模式覆盖**：仅覆盖"图N"、"第N页"、"fig.N"等常见模式，可能遗漏非常规表述
3. **boost权重固定**：0.3的boost在某些场景可能过大或过小，需要根据实际效果调优

### 5.2 后续优化方向

1. **重排模型增强**：引入跨模态重排模型，对图文chunk进行精细化排序
2. **查询改写**：对检测到图表引用的查询，用LLM改写为更利于检索的形式
3. **图嵌入检索**：将图描述生成嵌入向量，与查询进行专门的语义匹配
4. **多模态LLM直接看图**：将检索到的图片直接发送给多模态LLM，而非仅依赖文本描述

## 七、实际测试结果（Docker环境，真实大模型）

### 7.1 测试环境

- 部署方式：Docker（RAGFlow v0.25.6 镜像 + 代码补丁挂载）
- 知识库：工单15（含 CN100347506C.pdf，14个chunk，其中3个image类型含VLM图描述）
- 嵌入模型：text-embedding-v3@Tongyi-Qianwen（1024维）
- 对话大模型：mimo-v2.5-pro（OpenAI-API兼容）
- 检索参数：top_n=8，top_k=1024，similarity_threshold=0.2，vector_weight=0.3

### 7.2 关键发现：故障本质是"LLM幻觉"，而非"检索不到"

实测过程中发现，纯代码层优化（图表检测+定向召回+boost）**不足以达到100%准确率**。真正的故障表现为：

> **大模型编造部件名称。** 专利原文明确写"在保护管12里设置链条13...通过输入管14导入"，但LLM初版回答时把编号12说成"进料口"、编号13说成"分散室"、编号14说成"出料口"——这些名称在专利中根本不存在。

根因分析：图3的部件描述虽然已被检索到并送入上下文，但混杂在大段文字中。在缺乏强约束的情况下，LLM倾向于"脑补"一套看似合理的名称，而非严格依据原文。

**结论：跨模态检索优化必须"检索层 + 提示词层"双管齐下。** 检索层保证图纸信息进入上下文，提示词层强制LLM正确使用这些信息。

### 7.3 提示词迭代过程

| 版本 | 改进点 | 效果 |
|------|--------|------|
| 初版（系统默认） | 仅"总结知识库回答问题" | Q1-Q6大量编造部件名称，准确率约33% |
| 第1版 | 强约束：禁止编造名称、固定编号对应、引用空间描述词 | Q1-Q3正确，Q4答"入口处"（应为顶部） |
| 第2版 | 新增：方位优先采用[图表描述]中的方位词 | Q4仍纠结"图2非图3"而答"未找到" |
| **第3版（最终）** | 新增：图3=第11页、图2/图3编号体系相同可互用、散料流动顺序明示 | **Q1-Q6全部正确，准确率100%** |

```

【回答要求】
先定位[图表描述]中的关键句子，再据此简洁作答，并引用知识库内容[ID:n]。
```
